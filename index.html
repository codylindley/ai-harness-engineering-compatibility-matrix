Always-on instructions
AGENTS.mdroot of repo
 Yes — Yes Yes
The cross-tool universal standard. A markdown file at your repo root that any AI coding agent reads at the start of every session. Think of it as a system prompt for your repo — stack declarations, naming conventions, architectural boundaries, "never do X" rules. Governed by the Linux Foundation's Agentic AI Foundation. Put your shared team contract here. It has the broadest tool support of any config file, though Claude Code doesn't read it natively yet (open feature request).
CLAUDE.mdroot + subdirectories
 Partial1 Yes Fallback —
Claude Code's native config. Same purpose as AGENTS.md but Claude-specific. The standout feature is subdirectory scoping — drop a CLAUDE.md in /api and those rules only apply when Claude works on API files. If you also use AGENTS.md, your CLAUDE.md can simply say See @AGENTS.md for base rules plus Claude-specific additions. Copilot CLI reads it natively; VS Code requires the chat.useClaudeMdFile setting; Cloud agent does not support it.
copilot-instructions.md.github/
 Yes — — —
Copilot's original repo-wide config. Lives at .github/copilot-instructions.md. Auto-injected into every Copilot request. Same purpose as AGENTS.md but Copilot-exclusive. If you already have an AGENTS.md, you may not need this — or it can hold Copilot-only overrides. Auto-generate one by running /init in Copilot chat.
*.instructions.md.github/instructions/ (glob scoped)
 Yes — — —
Surgically scoped rules via glob patterns. Each file has an applyTo property in frontmatter (e.g., applyTo: "**/*.tsx") so rules only activate when working on matching files. Perfect for framework-specific conventions: solidjs.instructions.md for TSX, test.instructions.md for test files. Copilot auto-detects relevance. Claude Code's equivalent is .claude/rules/*.md with a paths property.
.claude/rules/*.mdscoped via paths property
 Partial2 Yes Fallback —
Claude Code's scoped instruction files. The Claude-side equivalent of Copilot's *.instructions.md. Each file uses a paths array in frontmatter (e.g., paths: ["**/*.ts"]) instead of Copilot's applyTo. Defaults to ** (all files) when omitted. VS Code Copilot can read these with the Claude file settings enabled; CLI and Cloud agent do not.
Custom agents (persona + tool restrictions)
*.agent.md.github/agents/
 Yes — — —
Specialized Copilot personas you invoke by name. Each file defines who answers — a distinct role with tool access, model choice, and behavioral boundaries. Invoke via @agent-name in chat. Examples: @reviewer (read-only), @test-writer (writes to /tests/ only). Can be used as subagents. Frontmatter controls tool and model access.
.claude/agents/*.mdClaude subagents
 — Yes Fallback —
Claude Code's subagent system. Specialized agents that Claude delegates work to, each running in isolated context. Built-in agents include Explore (read-only codebase research) and Plan (no file modifications). Skills can reference agents via the agent frontmatter field to run in a forked context.
.opencode/agents/*.mdOpenCode agents
 — — Yes —
OpenCode's native agent definitions. Same concept — persona with description, model, tool permissions, and prompt. Ships with built-in Build (full tool access) and Plan (read-only). Switch between primary agents with Tab. Can be defined in opencode.json or as markdown files.
Skills (bundled capability folders with SKILL.md)
.github/skills/*/SKILL.mdCopilot native path
 Yes — — —
Copilot's native skill path — Copilot-only. Skills are folders with a SKILL.md entrypoint plus optional scripts, templates, and reference docs. Copilot uses progressive disclosure: reads only the name + description (cheap), loads full body + resources only when relevant. Invoke via /skill-name or let Copilot auto-detect. Claude Code does not read this path — use .claude/skills/ for cross-tool compatibility.
.claude/skills/*/SKILL.mdClaude Code native path
 Yes Yes Fallback —
The best cross-tool skill path available today. Claude Code's native location, and Copilot auto-discovers it too. Put skills here to span both ecosystems without symlinks. Supports the full SKILL.md spec: frontmatter with name/description, markdown body, and bundled assets. Claude Code supports auto-invocation, user-invocation (/skill-name), and forked subagent execution.
.agents/skills/*/SKILL.mdcross-agent standard path
 — — Yes —
An emerging cross-agent skill path. Currently only read by OpenCode. Intended as a tool-agnostic location for skills, mirroring how AGENTS.md is a universal instruction file. May gain broader adoption as the Agent Skills standard matures.
.opencode/skills/*/SKILL.mdOpenCode native path
 — — Yes —
OpenCode's own skill directory. Same SKILL.md format. OpenCode also reads .claude/skills/ and .agents/skills/ as fallbacks, so you rarely need this unless you want OpenCode-exclusive skills.
Prompt files (manual /slash-command invocation)
*.prompt.md.github/prompts/
 Partial3 — — —
Lightweight reusable prompt templates. Invoke via /prompt-name in chat. No persona, no tool restrictions, no bundled assets — just a well-crafted prompt with optional frontmatter. Best for tasks you repeat: /new-component, /spell-check, /prep-pr. Can reference workspace files via markdown links. Only supported in VS Code and JetBrains IDEs — not Copilot CLI or Cloud agent. These don't port to any other tool.
.claude/commands/*.mdmerged into skills system
 — Yes Fallback —
Claude Code's original slash commands — now merged into skills. A command at .claude/commands/deploy.md and a skill at .claude/skills/deploy/SKILL.md both create /deploy and work identically. Existing command files keep working. Skills add optional supporting files, auto-invocation via description matching, and forked subagent execution.
Automation (lifecycle hooks)
.github/hooks/hooks.json
 Partial4 — — —
Copilot lifecycle hooks. Automated workflows triggered at agent events — run a formatter after every edit, validate branch names, enforce lint before commits. Think of them as git hooks for the AI agent loop. Supported in VS Code and CLI only — not Cloud agent.
.claude/hooks/settings.json hooks
 — Yes — —
Claude Code's hook system. 14+ trigger points including SessionStart, PreToolUse, PostToolUse, and PermissionRequest. Run custom scripts at every stage of the agent loop — linting, logging, approval gates. More granular than Copilot hooks, with triggers for both before and after tool execution.

1 CLAUDE.md in Copilot: CLI reads it natively at repo root. VS Code requires the chat.useClaudeMdFile setting. Cloud agent does not support it.

2 .claude/rules in Copilot: VS Code only, requires opt-in setting. Not supported in CLI or Cloud agent.

3 Prompt files: VS Code and JetBrains IDEs only. Not supported in Copilot CLI (open feature request) or Cloud agent.

4 Copilot hooks: VS Code and CLI only. Not supported in Cloud agent.

About tool surfaces: The Copilot column covers VS Code, CLI, and Cloud agent — they share almost all config support, with footnoted exceptions above. Claude Code is a single engine across CLI, VS Code extension, JetBrains, Desktop (including Cowork), and browser — all surfaces read the same config. Claude.ai chat (the consumer chat product) is a separate product and reads none of these files.
Key takeaways

AGENTS.md has the broadest cross-tool reach (3 of 4 tools), but Claude Code still doesn't read it natively — that's an open feature request.

The only skill path that spans both Copilot and Claude Code is .claude/skills/ — Copilot auto-discovers it. This eliminates the need for symlinks.

Prompt files (.prompt.md) are IDE-only. Claude Code's equivalent (.claude/commands/) has been merged into its skills system.

OpenCode has the best cross-compatibility story — reads AGENTS.md natively and falls back to Claude Code's paths for skills, agents, and commands.

Researched April 2026. This landscape changes fast — verify against current docs before committing to a config architecture.
